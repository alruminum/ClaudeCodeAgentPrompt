## Phase 0 — 컨텍스트 수집 + 캔버스 준비

### 목적
디자인 작업에 필요한 모든 정보를 수집하고, Pencil 캔버스에 작업 기반을 마련한다.

### 입력
- `@MODE:DESIGNER:DEFAULT` 또는 `@MODE:DESIGNER:UX_REDESIGN`
- `@PARAMS: { issue, screen, context }`

### 단계

1. **프로젝트 컨텍스트 읽기**
   - CLAUDE.md, impl 파일, 기존 디자인 토큰 (CSS 변수)
   - 대상 화면의 현재 코드 (App.tsx 등) — 읽기 전용 참조

2. **레퍼런스 리서치** (기존 Phase 0 유지)
   - WebSearch/WebFetch로 플랫폼 가이드라인, 경쟁사 레퍼런스 수집
   - 최소 3개 레퍼런스 확보

3. **Pencil 캔버스 준비**
   - Pencil MCP `batch_get`으로 기존 .pen 파일 확인
   - 기존 화면이 .pen으로 있으면 → 로드
   - 없으면 → 현재 앱 스크린샷 또는 Figma 가져오기로 베이스라인 생성
   - `get_screenshot`으로 현재 상태 캡처 → 베이스라인 기록

### 출력
- 레퍼런스 리서치 결과
- 캔버스에 베이스라인 화면 로드 완료

## Phase 1 — 변형 3개 생성

### 목적
Pencil 캔버스 위에 시각적으로 구분되는 3개 디자인 변형을 생성한다.

### 단계

1. **프레임 3개 생성**
   - Pencil MCP `batch_design`으로 별도 프레임 생성
   - 프레임 이름: `variant-A`, `variant-B`, `variant-C`
   - 각 프레임은 대상 화면의 완전한 디자인 (부분이 아닌 전체)

2. **차별화 규칙** (기존 유지)
   - 레이아웃 구조 (카드 vs 몰입형 vs 리스트)
   - 색상 팔레트 (톤, 채도, 온도)
   - 타이포그래피 (세리프 vs 산세리프 vs 디스플레이)
   - 인터랙션 강조 (Phase 4 애니메이션 스펙으로 표현)
   - **4축 중 2축 이상 차이 필수** — 색상만 다르면 1개로 취급

3. **차별화 검증 테이블 작성**

   ```
   | 축              | variant-A       | variant-B       | variant-C       |
   |-----------------|-----------------|-----------------|-----------------|
   | 레이아웃        | 카드 그리드     | 풀스크린 몰입형 | 수직 리스트     |
   | 색상 팔레트     | 다크 골드       | 다크 골드       | 네온 그린       |
   | 타이포그래피    | Barlow Condensed| Space Grotesk   | Barlow Condensed|
   | 인터랙션 강조   | 미니멀 트랜지션 | 3D 회전         | 스크롤 연동     |
   | 차이 축 수      | -               | 3축 차이 ✓      | 3축 차이 ✓      |
   ```

4. **스크린샷 캡처**
   - 각 프레임에 대해 `get_screenshot` 실행
   - 캡처 이미지는 Design-Critic에게 전달할 자료

5. **애니메이션 스펙 명시** (신규)
   - 각 변형에 대해 텍스트로 애니메이션 의도를 기술
   - 예: "variant-A: 버튼 호버 시 0.2s scale(1.05), 페이지 진입 시 카드 stagger fade-in 0.1s 간격"
   - 이 스펙은 Phase 4 코드 생성 시 구현 지침이 됨

### 출력
- `DESIGN_READY_FOR_REVIEW`
- Pencil 캔버스에 3개 프레임
- 3개 스크린샷 이미지
- 차별화 검증 테이블
- 변형별 애니메이션 스펙

## Phase 2 — Design-Critic 평가

### 목적
3개 변형을 객관적 기준으로 평가하고, 사용자 선택 단계로 진행할지 결정한다.

### 입력
- `@MODE:CRITIC:REVIEW`
- 3개 스크린샷 + 차별화 검증 테이블 + 애니메이션 스펙
- 대상 화면의 컨텍스트 (용도, 타겟 유저, 플랫폼)

### 평가 기준 (4축 × 10점 = 40점 만점)

1. **UX 명확성** (10점)
   - 흐름 직관성, 버튼 위계, 정보 밀도, 터치 친화성

2. **미적 독창성** (10점)
   - AI 클리셰 회피, 기억에 남는 요소, 대담한 선택
   - ※ Pencil 캔버스 기반이므로 실제 렌더링 품질로 평가

3. **컨텍스트 적합성** (10점)
   - 모바일 최적화, 목적 부합, 타겟 유저, 플랫폼 규약

4. **구현 가능성** (10점)
   - 금지 의존성 없음, 애니메이션 스펙의 현실성, 코드 구조 예측 가능성, 접근성
   - ※ 변경: "HTML 프리뷰 코드 품질" → "Pencil 디자인의 코드 변환 용이성"

### 판정

- **PICK** (≥30점, 모든 축 ≥5점) → Phase 3 사용자 선택으로
- **ITERATE** (피드백 포함) → Phase 1 재진입 (최대 3회)
  - ITERATE 시 이전 피드백 누적 전달
- **ESCALATE** (3회 반복 후에도 PICK 없음) → 사용자 직접 선택

### 출력
- 축별 점수 + 근거
- 판정 마커 (PICK / ITERATE / ESCALATE)
- ITERATE 시: 구체적 개선 피드백

## Phase 3 — 사용자 선택 + 미세조정

### 목적
사용자가 변형을 확인하고 선택하며, 필요시 직접 수정한다.

### 진행 방식

1. **하네스가 안내 메시지 출력**
   ```
   ✅ Design-Critic PICK — 3개 변형이 준비됐습니다.
   
   Pencil 캔버스에서 확인하세요:
   - variant-A: [한 줄 설명]
   - variant-B: [한 줄 설명]  
   - variant-C: [한 줄 설명]
   
   Pencil에서 확인/수정 후 선택할 변형을 입력하세요 (A/B/C):
   ```

2. **사용자가 Pencil 캔버스에서 확인**
   - 3개 프레임을 시각적으로 비교
   - 마음에 드는 변형을 선택
   - 필요시 캔버스에서 직접 수정 (색상, 위치, 크기, 텍스트 등)

3. **터미널에 선택 입력**
   - 사용자가 A/B/C 입력
   - 하네스가 선택된 프레임을 확정

### 출력
- 선택된 변형 프레임 이름
- 사용자 수정이 반영된 최종 .pen 상태

## Phase 4 — 코드 생성 (DESIGN_HANDOFF)

### 목적
확정된 디자인을 프로덕션 코드로 변환한다.

### 단계

1. **확정 디자인 읽기**
   - Pencil MCP로 선택된 프레임의 최종 상태 읽기
   - `batch_get`으로 전체 요소 구조, 스타일, 변수 추출
   - `get_screenshot`으로 최종 스크린샷 캡처 (코드 검증 기준용)

2. **DESIGN_HANDOFF 패키지 생성**

   a. **디자인 토큰 테이블**
   ```
   | 토큰               | 값              | CSS 변수              |
   |--------------------|----------------|-----------------------|
   | primary-color      | #D4A843        | --vb-accent           |
   | surface-bg         | #18181b        | --vb-surface          |
   | font-score         | Barlow Condensed| --vb-font-score      |
   ```

   b. **컴포넌트 구조**
   - 컴포넌트 트리 (부모-자식 관계)
   - 각 컴포넌트의 props 인터페이스 (View-Only: 더미 데이터)

   c. **애니메이션 구현 스펙**
   - Phase 1에서 명시한 애니메이션 스펙을 CSS keyframes/transition으로 구체화
   - 예시 코드 포함

   d. **엔지니어 노트**
   - 구현 시 주의사항
   - 기존 코드와의 충돌 가능성
   - 성능 고려사항

3. **코드 생성** (Pencil AI Chat 또는 Claude 직접)
   - 프로젝트 설정에 맞는 프레임워크 (React/Vue/등)
   - View-Only 원칙: 더미 데이터, 스토어/훅 접근 금지
   - 출력 위치: `design-variants/` 디렉토리 (src/ 직접 수정 금지)

### 출력
- `DESIGN_HANDOFF`
- 디자인 토큰 테이블
- 컴포넌트 구조 문서
- 애니메이션 구현 스펙
- 코드 파일 (design-variants/ 하위)
- 엔지니어 노트
