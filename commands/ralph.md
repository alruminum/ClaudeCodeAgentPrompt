# Ralph Wrapper Skill

유저의 자연어 요청을 구조화된 Ralph Loop 프롬프트로 변환한다.

## 입력 파싱

유저 메시지에서 다음을 추출한다:
- **반복 횟수**: "N번", "N회", "N iterations", "N번만" → `--max-iterations N` (없으면 기본 3)
- **작업 내용**: 나머지 전체 텍스트

## 구조화 규칙

아래 템플릿으로 프롬프트를 작성한다.
작업 내용이 모호하더라도 최대한 의도를 추론해서 채운다. 추측이 불가능한 항목만 `[미정]`으로 남긴다.

### 핵심 원칙: 계획 파일을 "따라하라"고 주지 않는다

계획 파일을 그대로 따르면 기계적 찾아바꾸기만 하고 맥락을 이해하지 않으며, 계획에 빠진 항목은 영원히 놓친다.
프롬프트는 3단 구조로 작성한다:

1. **목표와 설계를 프롬프트 본문에 직접 기술** — 핵심 설계(왜 바꾸는지, 새 구조가 뭔지)를 인라인. "계획 파일 읽고 따라하라"가 아니라 "이 목표를 달성하라"
2. **계획 파일은 완료도 체크리스트로만 참조** — 매 회차 끝에 "계획 파일 대조해서 빠진 거 없나 확인"
3. **잔여 참조 grep 스캔 강제** — 매 회차 끝에 옛 키워드(변경 전 이름들)를 grep해서 계획에 없던 누락도 잡음

```
## 목표
[작업 내용에서 핵심 목표를 1~3문장으로 명확히 기술. 구체적일수록 좋다.]

## 설계 (목표 달성을 위한 핵심 구조/규칙)
[새 구조, 변경 이유, 주요 규칙을 프롬프트에 직접 기술. 계획 파일 참조가 아니라 인라인으로.]

## 제약사항
[작업 내용에서 언급된 제약. 없으면 일반 제약 추론:
- 기존 코드/파일 구조 유지
- 외부 의존성 추가 금지 (명시 없을 시)
- 등]

## 컨텍스트 관리
매 회차 시작 전 `/clear`로 컨텍스트를 클리어한다.
따라서:
- 매 회차 시작 시 progress 파일(`/tmp/ralph_{slug}_progress.md`)을 읽어 현재 상태 파악
- 이전 회차 기억에 의존하지 않는다. 파일을 직접 읽어 "이미 끝난 것"과 "아직 남은 것"을 판별
- 매 회차 종료 시 progress 파일에 완료 항목과 남은 항목을 기록

## 최소 2회차 강제 규칙
- **1회차**: 작업 수행 (구현/수정). 1회차에서는 절대 완료 선언하지 않는다.
- **2회차 이후**: 이전 회차 결과를 검증. 검증 통과 시에만 완료 가능.
- progress 파일에 `iteration: N` 기록. N < 2이면 완료 선언 금지.

## 매 회차 종료 시 검증 루틴
1. **체크리스트 대조**: [계획 파일 경로]를 읽어 빠진 항목이 없는지 확인
2. **잔여 참조 grep**: `grep -r "[옛 키워드들]" [대상 경로]`로 변경 전 이름이 남아있는지 스캔
3. **자가 질문**:
   - 이번 회차에서 수정하거나 발견한 문제가 있었는가?
     → 있었다면: 다음 회차에서 파급 효과 확인
   - 대상 전체를 다시 훑었을 때 아직 손대지 않은 부분이 남아 있는가?
   - 지금 멈춰도 목표가 충분히 달성됐다고 확신할 수 있는가?

## 적대적 교차 검증 (완료 선언 직전 필수)

"내가 반드시 빵꾸를 냈을 것이다"는 전제로, 아래 4가지를 반드시 실행한다.
**이 검증을 건너뛰고 완료 선언하는 것은 금지.**

### A. 코드 블록 실존 검증
문서(.md) 안의 코드 블록에서 참조하는 함수명, CLI 파라미터, 파일 경로, 클래스명을 추출하고,
실제 코드에서 grep해서 존재하는지 확인한다.
- 예: 스킬 .md에 `executor.py --phase system-design`이 있으면 → executor.py argparse에 `--phase`가 실제로 있는지
- 예: .md에 `run_ux_validation()`이 있으면 → core.py에 def run_ux_validation이 있는지

### B. 주어-위치 교차 검증
문서에서 "X 루프에서 Y를 한다"는 문장의 **주어(X 루프)가 실제 코드 위치와 맞는지** 확인한다.
- 예: "plan 루프에서 architect(SD)를 호출한다" → plan_loop.py에 architect 호출이 있는지
- 예: "설계 루프에서 designer를 호출한다" → 메인 Claude 오케스트레이션인데 harness 코드에 있으면 모순

### C. 문서 간 수치/이름 일치
마커 이름, 타임아웃 값, 에이전트 이름 등이 참조하는 모든 문서에서 동일한지 확인한다.
- 예: orchestration-rules.md의 마커 테이블 vs core.py Marker enum vs 에이전트 .md
- 예: 타임아웃 600s가 orchestration-rules.md와 plan_loop.py에서 동일한지

### D. 엣지케이스 시나리오 워크스루
변경된 코드 경로에 대해 최소 2개 엣지케이스를 상상하고 코드를 따라간다.
- 예: "ux-flow.md는 있는데 prd.md가 없으면?" → plan_loop.py에서 어떤 경로를 타는지
- 예: "ux-architect가 마커 없이 끝나면?" → UNKNOWN → UX_FLOW_ESCALATE 처리되는지

**A~D 중 하나라도 불일치를 발견하면 완료 선언 금지. 수정 후 재검증.**

**완료 조건** (모두 충족해야 함):
- 최소 2회차 이상 진행됨 (progress 파일의 iteration >= 2)
- 세 자가 질문 모두 "문제 없음 / 남은 것 없음 / 확신함"
- 적대적 교차 검증 A~D 전부 통과 (progress 파일에 각 항목 결과 기록)
- 위 조건 충족 시 아래를 출력:
<promise>RALPH_DONE</promise>

조건 미충족 시 절대 출력하지 않는다. 루프를 탈출하기 위해 거짓으로 출력하는 것은 엄격히 금지된다.
```

## 실행 절차

### 1단계: 구조화된 프롬프트 생성 후 유저에게 제시

프롬프트를 생성한 뒤 아래 형식으로 유저에게 보여준다:

```
---
**Ralph Loop 실행 설정**

- 최대 반복: N회
- 완료 시 조기 탈출: 활성화 (RALPH_DONE 출력 시)

**구조화된 프롬프트:**

[생성된 프롬프트 전체 출력]

---
이대로 실행할까요? 수정이 필요하면 말씀해 주세요.
```

### 2단계: 유저 확인 대기

- 유저가 "응", "ㅇㅇ", "ok", "고", "실행", "그래" 등 긍정 응답 → 3단계 진행
- 유저가 수정 요청 → 해당 부분만 수정 후 다시 1단계 출력
- 유저가 취소 → 종료

### 3단계: 임시 파일 생성 후 ralph-loop 호출

프롬프트에 `[`, `]`, `(`, `)` 등 shell 특수문자가 포함될 수 있으므로,
**반드시** 임시 파일 방식으로 호출한다. 세션 충돌 방지를 위해 `mktemp`로 고유 파일명을 생성한다.

1. Bash 도구로 고유 파일명 생성 (macOS mktemp는 .md suffix 미지원):
```bash
echo "/tmp/ralph_task_$(date +%s)_$RANDOM.md"
```

2. Write 도구로 프롬프트를 위 경로에 저장 (예: `/tmp/ralph_task_1775794108_17464.md`)

3. Skill 도구로 ralph-loop 호출 (짧은 참조 프롬프트 사용):

```
Skill: ralph-loop:ralph-loop
Arguments: "Read <TMPFILE> and execute the task described there" --max-iterations N --completion-promise "RALPH_DONE"
```

4. 루프 완료(RALPH_DONE 또는 max-iterations 도달) 후 Bash로 임시 파일 삭제:

```bash
rm -f <TMPFILE>
```
