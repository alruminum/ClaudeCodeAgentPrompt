# QA Loop Skill

유저의 이슈 신고를 구조화된 리포트로 만들고, 부족한 정보는 역질문으로 채운 뒤 하네스 루프를 시작한다.

## 0단계: 버그 vs 디자인 구분

이슈 유형을 먼저 판단한다.

**명확히 동작 버그**: 기능이 작동 안 함, 로직 오류, 타이머/카운터 이상, 버튼 중복 클릭 등
→ **bugfix 루프**로 진행

**명확히 디자인 이슈**: "시안", "디자인", "UX", "화면 개선", "이터레이션", "모양 바꿔줘"
→ **design 루프**로 진행

**애매한 경우** ("이상해", "안 맞아", "칸이 이상해", "위치가 다른데"):
→ 아래 질문 1개만 한다:

> "동작이 잘못된 건가요(기능 버그), 아니면 보이는 게 마음에 안 드는 건가요(디자인 이슈)?"

유저 답변으로 루프를 결정한다.

## bugfix 루프 경로

### 정보 추출

| 항목 | 설명 | 예시 |
|------|------|------|
| **증상** (ACTUAL) | 지금 어떻게 동작하고 있는가 | "안눌러도 초가 무한으로 지나가" |
| **기대** (EXPECTED) | 어떻게 동작해야 하는가 | "2초 제한으로 끝나야 함" |
| **위치** (WHERE) | 어느 화면/기능/조건에서 발생하는가 | "게임 플레이 중 타이머" |

**증상(ACTUAL)이 없으면** 반드시 물어본다. 기대/위치는 추론 가능하면 넘어간다.
질문은 한 번에 최대 2개.

### 확인 및 실행

```
---
**QA Loop 실행 설정** (bugfix)

[증상] ...
[기대] ...
[위치] ...

이대로 bugfix 루프 시작할까요?
---
```

확인 후 Bash 실행:
```bash
bash ~/.claude/harness/executor.sh bugfix \
  --bug "[증상] <실제동작> / [기대] <기대동작> / [위치] <위치>" \
  --prefix mb
```

## design 루프 경로

### 정보 추출

| 항목 | 설명 | 예시 |
|------|------|------|
| **화면/컴포넌트** (WHERE) | 어느 화면인가 | "게임 메인 화면" |
| **문제/요청** (WHAT) | 무엇이 이상하거나 어떻게 바꾸고 싶은가 | "칸이 안 맞아", "더 임팩트 있게" |
| **참고** (REF) | 스크린샷, 시안, 기준 있으면 | 이미지 첨부 등 |

화면/컴포넌트가 없으면 물어본다. 참고는 없어도 진행.

### 확인 및 실행

```
---
**QA Loop 실행 설정** (design)

[화면] ...
[요청] ...
[참고] ...

이대로 design 루프 시작할까요? (designer → design-critic → 유저 선택)
---
```

확인 후 Bash 실행:
```bash
bash ~/.claude/harness/executor.sh design \
  --context "[화면] <화면> / [요청] <요청>" \
  --prefix mb
```

GitHub 이슈 번호를 유저가 언급했으면 `--issue <N>` 추가.
